Every workflow is a linear sequence of steps. A step can be:

- A **trigger** — the entry point that fires the workflow
- A **pre-built action** — reusable code maintained by Pipedream for 2,800+ apps
- A **custom code step** — Node.js, Python, Go, or Bash you write directly

## Step naming

Each step has a unique name within the workflow (e.g., `trigger`, `get_user`, `send_notification`). The name is used to reference that step's output in downstream steps.

Rename a step by clicking its name in the workflow editor. Valid names use lowercase letters, numbers, and underscores.

## Accessing trigger data

The trigger is always named `trigger`. Its full event payload is available at `steps.trigger.event`.

<Tabs>
  <Tab title="Node.js">
    ```javascript
    export default defineComponent({
      async run({ steps, $ }) {
        // HTTP trigger: body, headers, method, path, query
        const body    = steps.trigger.event.body;
        const headers = steps.trigger.event.headers;
        const method  = steps.trigger.event.method;

        // App event trigger (e.g., GitHub)
        const issue = steps.trigger.event;
        console.log(issue.title, issue.html_url);
      },
    });
    ```
  </Tab>
  <Tab title="Python">
    ```python
    def handler(pd: "pipedream"):
        body    = pd.steps["trigger"]["event"]["body"]
        headers = pd.steps["trigger"]["event"]["headers"]

        # App event trigger
        issue = pd.steps["trigger"]["event"]
        print(issue["title"], issue["html_url"])
    ```
  </Tab>
</Tabs>

## Exporting data from a step

A step's return value is automatically exported as `$return_value` and available to all later steps.

<Tabs>
  <Tab title="Node.js">
    ```javascript
    export default defineComponent({
      async run({ steps, $ }) {
        const response = await fetch(
          `https://api.github.com/repos/${steps.trigger.event.body.repo}`
        );
        const repo = await response.json();

        // Returned value is steps.get_repo.$return_value in later steps
        return {
          fullName:    repo.full_name,
          stars:       repo.stargazers_count,
          openIssues:  repo.open_issues_count,
          language:    repo.language,
        };
      },
    });
    ```
  </Tab>
  <Tab title="Python">
    ```python
    import requests

    def handler(pd: "pipedream"):
        repo_slug = pd.steps["trigger"]["event"]["body"]["repo"]
        r = requests.get(f"https://api.github.com/repos/{repo_slug}")
        repo = r.json()

        # Returned dict is pd.steps["get_repo"]["$return_value"] in later steps
        return {
            "full_name":   repo["full_name"],
            "stars":       repo["stargazers_count"],
            "open_issues": repo["open_issues_count"],
        }
    ```
  </Tab>
</Tabs>

### Named exports (Node.js)

Use `$.export(name, value)` to export multiple named values from a single step.

```javascript
export default defineComponent({
  async run({ steps, $ }) {
    const response = await fetch("https://api.example.com/orders?status=pending");
    const data = await response.json();

    // Export named values — each is addressable downstream
    $.export("total",    data.total);
    $.export("orders",   data.orders);
    $.export("currency", data.currency);

    // Also return a summary as $return_value
    return { count: data.orders.length };
  },
});
```

Access named exports in a downstream step:

```javascript
// steps.fetch_orders.total
// steps.fetch_orders.orders
// steps.fetch_orders.currency
// steps.fetch_orders.$return_value.count
const orders   = steps.fetch_orders.orders;
const total    = steps.fetch_orders.total;
```

## Referencing prior step data

| What you want | Node.js | Python |
|---|---|---|
| Trigger event | `steps.trigger.event` | `pd.steps["trigger"]["event"]` |
| Step return value | `steps.<name>.$return_value` | `pd.steps["<name>"]["$return_value"]` |
| Named export | `steps.<name>.<export>` | `pd.steps["<name>"]["<export>"]` |

## Step props

Props are typed, configurable inputs declared on a step. Pipedream renders them as form fields in the workflow editor and validates them before running the step.

```javascript
export default defineComponent({
  props: {
    // Connected app — Pipedream injects auth
    github: { type: "app", app: "github" },

    // String input shown in the editor
    repo: {
      type: "string",
      label: "Repository",
      description: "The full repo slug, e.g. acme/backend",
    },

    // Optional integer with default
    perPage: {
      type: "integer",
      label: "Results per page",
      default: 30,
      optional: true,
    },

    // Persistent key-value store
    data: { type: "data_store" },
  },
  async run({ steps, $ }) {
    // this.github.$auth.oauth_access_token
    // this.repo  => "acme/backend"
    // this.perPage => 30
    // this.data.get("key")
  },
});
```

## Controlling flow

### Exit early

Stop a workflow execution cleanly — without marking it as an error:

```javascript
export default defineComponent({
  async run({ steps, $ }) {
    const eventType = steps.trigger.event.body.type;

    // Ignore events we don't care about
    if (eventType !== "payment.succeeded") {
      $.flow.exit(`Skipping event type: ${eventType}`);
    }

    // Continue processing payment.succeeded events...
  },
});
```

### Error handling

Unhandled exceptions mark the step (and execution) as failed. Catch expected errors and handle them explicitly:

```javascript
export default defineComponent({
  async run({ steps, $ }) {
    let user;
    try {
      const res = await fetch(
        `https://api.example.com/users/${steps.trigger.event.body.userId}`
      );
      if (res.status === 404) {
        $.flow.exit("User not found — skipping");
      }
      user = await res.json();
    } catch (err) {
      throw new Error(`Failed to fetch user: ${err.message}`);
    }

    return user;
  },
});
```

## Step execution order

Steps always run top-to-bottom. A step cannot reference the output of a step that comes after it.

<Note>
  If you need conditional branching, use `$.flow.exit()` to short-circuit remaining steps, or split logic into separate workflows triggered via HTTP.
</Note>
